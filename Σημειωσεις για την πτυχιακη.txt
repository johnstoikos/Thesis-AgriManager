Σημειωσεις για την πτυχιακη

01/03/2026

1. Υποδομή Βάσης Δεδομένων (Database Layer)

    Σύστημα: PostgreSQL με το spatial extension PostGIS.

    Δημιουργία Βάσης: agrimanager_db.

    Ρύθμιση Πίνακα: Δημιουργήθηκε ο πίνακας fields με στήλη τύπου GEOMETRY(Polygon, 4326). Το SRID 4326 αντιστοιχεί στο παγκόσμιο σύστημα συντεταγμένων WGS 84 (GPS).

2. Τεχνολογίες Backend (Spring Boot Stack)

    Framework: Spring Boot 3.x.

    Διαχείριση Γεωχωρικών Δεδομένων: Χρήση της βιβλιοθήκης JTS (Java Topology Suite) για τον χειρισμό γεωμετριών (Polygons) μέσα στη Java.

    Serialization/Deserialization: Προσθήκη του jackson-datatype-jts στο pom.xml και παραμετροποίηση μέσω JacksonConfig, ώστε το API να μπορεί να δέχεται και να στέλνει δεδομένα σε μορφή GeoJSON.

3. Αρχιτεκτονική Κώδικα (N-Tier Architecture)

Υλοποιήθηκε η δομή "Layered Architecture" για καθαρότητα και επεκτασιμότητα:

    Model (Entity): Η κλάση Field.java. Αντιπροσωπεύει τον πίνακα της βάσης. Χρησιμοποιεί το annotation @Entity και το πεδίο Polygon boundary (JTS).

    DTO (Data Transfer Object): Η κλάση FieldDTO.java. Χρησιμοποιείται για την ασφαλή μεταφορά δεδομένων μεταξύ Frontend και Backend, απομονώνοντας το database schema από το API.

    Repository: Το FieldRepository (Interface). Κληρονομεί από το JpaRepository, παρέχοντας έτοιμες μεθόδους CRUD (Save, Find, Delete) χωρίς συγγραφή SQL.

    Service Layer: Η κλάση FieldService.java. Εδώ βρίσκεται η "Business Logic". Υλοποιήθηκε αυτόματος υπολογισμός της έκτασης (area) του αγροτεμαχίου κατά την αποθήκευση, χρησιμοποιώντας τη μέθοδο .getArea() της JTS.

    Controller: Ο FieldController.java. Ορίζει τα REST Endpoints:

        GET /api/fields: Επιστρέφει τη λίστα όλων των αγροτεμαχίων σε JSON.

        POST /api/fields: Δέχεται GeoJSON, το αποθηκεύει και επιστρέφει το δημιουργημένο αντικείμενο με το ID του.

4. Εργαλεία Δοκιμών & Διαχείρισης

    DBeaver: Χρήση για τη διαχείριση της βάσης και την οπτικοποίηση των πολυγώνων πάνω σε χάρτη (Spatial View).

    cURL: Χρήση του τερματικού για τη διενέργεια HTTP POST requests με GeoJSON payloads για τον έλεγχο της σωστής λειτουργίας του API.

1. Γεωμετρία Καλλιέργειας (Crop Geometry)

    Polygon Mapping: Κάθε καλλιέργεια (Crop) απέκτησε πλέον το δικό της πολύγωνο (zoneBoundary). Αυτό επιτρέπει στο χρήστη να ορίσει πού ακριβώς μέσα στο χωράφι έχει φυτέψει τι (π.χ. το πάνω μισό ελιές, το κάτω μισό σιτάρι).

    Entity & DTO Update: Ενημερώσαμε το Crop.java και το CropDTO.java ώστε να δέχονται και να αποθηκεύουν αντικείμενα τύπου Polygon από τη βιβλιοθήκη JTS.

2. Χωρικός Περιορισμός (Spatial Validation)

Υλοποιήσαμε τον πρώτο και πιο κρίσιμο γεωχωρικό έλεγχο στο CropService:

    Validation Check: field.getBoundary().contains(crop.getZoneBoundary()).

    Λειτουργία: Το σύστημα πλέον απαγορεύει την αποθήκευση μιας καλλιέργειας αν το πολύγωνό της βγαίνει έστω και λίγο έξω από τα όρια του χωραφιού (Field).

    Error Trigger: Σε περίπτωση παράβασης των ορίων, η εφαρμογή "πετάει" RuntimeException με το μήνυμα: "Η ζώνη καλλιέργειας είναι εκτός των ορίων του χωραφιού!".

3. Συσχέτιση Δεδομένων (Relational Mapping)

    Many-to-One: Επιβεβαιώσαμε τη σύνδεση της καλλιέργειας με το χωράφι (field_id), επιτρέποντας σε ένα χωράφι να φιλοξενεί πολλαπλά διαφορετικά Crops (πολυκαλλιέργεια).

4. Οπτικοποίηση (Spatial View)

    DBeaver Integration: Μέσω SQL UNION ALL είδαμε για πρώτη φορά στο Spatial View το "πάντρεμα" των δύο γεωμετριών: Το μεγάλο πολύγωνο του χωραφιού και μέσα του το μικρότερο πολύγωνο της καλλιέργειας.
    
    1. Ολοκλήρωση της Ιεραρχίας Δεδομένων (Tasks)

Προσθέσαμε το τελευταίο επίπεδο πληροφορίας στην αγροτική διαχείριση:

    Task Entity & DTO: Δημιουργήσαμε το μοντέλο των Εργασιών (Πότισμα, Λίπανση, Ψεκασμός).

    Point Geometry: Κάθε εργασία δεν έχει μόνο ημερομηνία και περιγραφή, αλλά και ένα γεωγραφικό σημείο (Point) στο χάρτη.

    Relationship: Συνδέσαμε τις εργασίες με το Crop (Many-to-One), ολοκληρώνοντας τη δομή: Field ⮕ Crop ⮕ Task.

2. Γεωχωρικός Έλεγχος Ακριβείας (Point-in-Polygon)

Αναπτύξαμε "έξυπνη" λογική στο TaskService:

    Spatial Validation: Προσθέσαμε έλεγχο ώστε το σημείο (Point) της εργασίας να βρίσκεται υποχρεωτικά μέσα στο πολύγωνο (Polygon) της ζώνης καλλιέργειας.

    Προστασία Δεδομένων: Αν ένας χρήστης πάει να καταχωρήσει έναν ψεκασμό σε συντεταγμένες εκτός του Crop του, το σύστημα απορρίπτει την εγγραφή με RuntimeException.

3. Επαγγελματική Διαχείριση Σφαλμάτων (Error Handling)

Μετατρέψαμε το API από "πειραματικό" σε "παραγωγικό":

    Global Exception Handler: Δημιουργήσαμε μια κεντρική κλάση (@ControllerAdvice) που "πιάνει" τα σφάλματα της Java.

    Custom API Responses: Αντί για το ακαταλαβίστικο "500 Internal Server Error", το API πλέον απαντάει με ένα καθαρό JSON που περιέχει:

        Το ακριβές μήνυμα του σφάλματος (π.χ. "Εκτός ορίων").

        Το HTTP Status Code (400 Bad Request).

        Το Timestamp του συμβάντος.

4. Έλεγχος & Επιβεβαίωση (Testing)

    CURL Tests: Δοκιμάσαμε με επιτυχία σενάρια "Επιτυχίας" και "Αποτυχίας" για την εισαγωγή Tasks.

    DBeaver Spatial View: Επιβεβαιώσαμε οπτικά ότι οι "πινέζες" των Tasks εμφανίζονται σωστά πάνω στα πολύγωνα των Crops στο χάρτη.
